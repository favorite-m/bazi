# 客户档案管理规范 (Customer Profile Management Standard)

## 0. 核心理念

- **中文命名**：所有客户档案文件使用中文命名，附带数字前缀控制加载顺序
- **按需加载**：大模型不一次性读取所有文件，通过脚本查询 + 目录索引决定加载范围
- **脚本驱动**：目录的维护和客户信息查询统一通过 `memory-retriever.py` 执行
- **上下文干净**：严禁将完整的客户档案全部注入上下文，仅加载当前分析所需的最小文件集

## 1. 目录结构

```
/客户档案/
├── 客户档案总目录.md       ← 全局主索引（由脚本自动维护）
│
├── {用户称呼}/
│   ├── 0-目录.md           ← 本客户档案目录与加载策略（必需）
│   ├── 0-基础信息.md       ← 基础信息、命理数据（必需）
│   ├── 1-分析记录日志.md   ← 历次分析记录时间线（必需）
│   ├── 2-八字分析.md       ← 八字排盘与格局分析（可选）
│   ├── 3-紫微斗数.md       ← 紫微斗数命盘（可选）
│   ├── 4-事业财运.md       ← 事业财运分析（可选）
│   ├── 5-感情婚姻.md       ← 感情婚姻分析（可选）
│   ├── 6-生活维度.md       ← 田宅、福德、健康等（可选）
│   ├── 7-流年预测.md       ← 流年/大运详批（可选）
│   ├── 8-特殊记录.md       ← 特殊记录（可选）

```

## 2. 文件命名规范

### 2.1 数字前缀规则

| 前缀 | 命名规则 | 文件类型 | 必需 |
|:----:|:---------|:---------|:----:|
| `0-` | `0-目录.md` | 本客户档案目录 | ✅ |
| `0-` | `0-基础信息.md` | 基础信息与命理数据 | ✅ |
| `1-` | `1-分析记录日志.md` | 历次分析记录 | ✅ |
| `2-` | `2-八字分析.md` | 八字命盘与格局 | |
| `3-` | `3-紫微斗数.md` | 紫微斗数命盘 | |
| `4-` | `4-事业财运.md` | 事业财运分析 | |
| `5-` | `5-感情婚姻.md` | 感情婚姻分析 | |
| `6-` | `6-生活维度.md` | 田宅、福德、健康等 | |
| `7-` | `7-流年预测.md` | 流年/大运详批 | |
| `8-` | `8-特殊记录.md` | 特殊记录 | |
| `9-` | `9-梦境解析.md` | 梦境解析 | |

### 2.2 命名原则

- **必须使用中文命名**（客户是中文用户，档案文件理应中文）
- **数字前缀**控制排序和加载优先级，`0-` 为最优先
- 同一前缀数字内按字母序排列
- 后续新增文件按业务相关性插入对应的数字区间

## 3. 客户档案总目录（全局索引）

文件位置：`/客户档案/客户档案总目录.md`

**用途**：快速检索是否存在某客户，以及该客户的基本命理特征

**格式**：
```markdown
# 客户档案总目录

> 本文件由 `memory-retriever.py` 脚本自动维护更新

---

## 客户清单

| 序号 | 用户 | 文件夹 | 日主 | 格局 | 分析次数 | 最近更新 | 操作指引 |
|:----:|:----|:------|:----:|:----:|:--------:|:--------:|:---------|

---

## 快速检索

如需查找特定用户，调用 `memory-retriever.py`：
```bash
python "$SCRIPT_PATH" --query "客户名 命盘" --target-dir /客户档案
```
```

**维护方式**：
- 新增/删除客户时 → 调用 `memory-retriever.py` 重新扫描 `/客户档案/` 目录
- 大模型不直接编辑总目录，仅通过脚本查询
- 总目录的目标是**极小体积**（几行表格），方便快速扫读

## 4. 子目录索引（0-目录.md）

**用途**：记录本客户文件夹内的所有文件及其加载策略

**必须包含**：
- 完整的文件清单表格
- 按场景的加载策略（新对话/继续分析/专项分析）
- 文件的大小说明（精简/中/详）

**示例**：
```markdown
# {称呼} 命理分析档案 - 目录

> 阳历：... | 日主：... | 格局：...
> 创建：... | 最近更新：...

## 文件清单

| 编号 | 文件名 | 内容 | 大小 | 按需加载 |
|:----:|:------|:-----|:----:|:--------:|

## 加载策略

| 场景 | 加载文件 |
|:----|:---------|
| 新对话识别客户 | 先查总目录，再读 `0-基础信息.md` |
| 继续分析命盘 | `0-基础信息.md` + `1-分析记录日志.md` |
| 专项分析（事业） | 按需加载 `4-事业财运.md` |
```

## 5. 基础信息标准（0-基础信息.md）

文件顶部必须包含 YAML 元数据头：

```yaml
---
name: "用户称呼"
slug: "yonghu-pinyin"
gender: "M" | "F"
birth_solar: "YYYY-MM-DD HH:mm"
birth_lunar: "农历年月日 时辰"
birth_place: "省/市"
calendar_type: "gregorian" | "lunar"
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
tags: [客户, 命理标签1, 命理标签2]
---
```

## 6. 分析记录日志标准（1-分析记录日志.md）

每次分析后追加记录：

```markdown
## {YYYY-MM-DD} - {分析类型}

**分析摘要**：{一句话描述}

**触发原因**：{用户主动请求 / 流年变化 / 梦境 / 事件校验}

**关键发现**：
- {发现1}

**状态标记**：{完成 / 待跟进 / 喜用神需修正}
---
```

### 6.1 喜用神修正记录（特殊格式）

当喜用神判定需要修正时，必须在 `1-分析记录日志.md` 中使用以下标准格式记录：

```markdown
## {YYYY-MM-DD} - 喜用神修正

**触发**：前事校验发现冲突

**旧判定**：
- 喜用神：{旧喜用神}
- 忌神：{旧忌神}

**冲突证据**：
- {年份}年：{事件描述}（应为喜/忌神年份但实际吉/凶）

**新判定**：
- 喜用神：{新喜用神}
- 忌神：{新忌神}
- 修正依据：{修正理由}

**状态标记**：完成
---
```

修正后必须同步更新 `0-基础信息.md` 中的喜用神/忌神字段。

## 7. 新对话客户查找流程

```
Step 1: 读取 /客户档案/客户档案总目录.md（极小体积，快速扫读）
Step 2: 如果总目录中有匹配用户：
   ├─ 进入该用户的文件夹
   ├─ 读取 0-目录.md 了解文件结构和加载策略
   ├─ 读取 0-基础信息.md 获取基础命理数据
   ├─ 按需读取 1-分析记录日志.md 获取历史分析记录
   └─ 需要专项分析时，按 0-目录.md 的列表按需加载对应文件
Step 3: 如果总目录中无匹配用户：
   ├─ 调用 memory-retriever.py 进行语义搜索
   │   python "$SCRIPT_PATH" --query "{用户特征}" --target-dir /客户档案
   ├─ 如果找到 → 读取并继续
   └─ 如果未找到 → 询问用户是否新建档案
Step 4: 严禁一次性加载整个客户文件夹的所有文件
```

## 8. 更新协议

- **总目录更新**：新增/删除客户后，运行以下命令自动更新 `客户档案总目录.md`：
  ```bash
  python .reasonix/skills/documentation-management/scripts/update_catalog.py
  ```
- **分析记录追加**：每次分析结束后，追加到 `1-分析记录日志.md`
- **喜用神修正**：记录修正原因到 `1-分析记录日志.md`，更新 `0-基础信息.md`
- **0-目录.md 更新**：文件增删时同步更新
