# Code Header Comment Skill

## 角色 (Role)
模仿一名严谨的文档工程师 (Documentation Engineer)。你擅长从代码中提取核心信息，并将其组织成清晰、规范的头部注释。

## 适用场景 (Trigger)
*   **生成新代码时**: 必须为新创建的文件生成头部注释。
*   **重构旧代码时**: 如果旧文件缺乏注释，需要补充。
*   **代码审查时**: 检查现有注释是否完整、准确。

## 规范 (Standards)

### 1. 头部注释结构 (Header Structure)
每个文件必须包含以下字段：
```python
# -*- coding: utf-8 -*-
"""
[文件名]: [简短描述]
[功能]: [详细描述文件功能]
[创建日期]: YYYY-MM-DD
[关键修改历史]:
    - YYYY-MM-DD: [修改内容]
[依赖]: [列出关键依赖模块]
[输入]: [输入数据或参数]
[输出]: [输出数据或返回值]
"""
```

### 2. 字段说明 (Field Descriptions)
*   **文件名**: 文件名 (e.g., `main.py`)
*   **功能**: 1-2句话描述文件核心功能
*   **创建日期**: 代码首次创建日期
*   **关键修改历史**: 每次修改的日期、作者和简短描述
*   **依赖**: 关键的 import 模块或外部库
*   **输入**: 函数/类接收的参数或数据源
*   **输出**: 函数/类返回的值或产生的效果

### 3. 语言要求
*   注释内容使用**中文**
*   日期格式为 YYYY-MM-DD
*   代码行首使用 `# -*- coding: utf-8 -*-"` 声明编码

## 检查清单 (Checklist)
- [ ] 是否包含所有标准字段？
- [ ] 内容是否准确反映代码实际功能？
- [ ] 日期格式是否正确？
- [ ] 语言是否为中文？
- [ ] 是否有死代码 (Dead Code)？
- [ ] 所有的 `TODO` 是否都有明确的后续计划？

**核心原则**： Code Header Comment 首先是服务于 Agent 的，其次才是服务于人类的。所以，Code Header Comment 应该包含 Agent 快速理解文件用途、职责、输入输出，并判断是否需要深入阅读代码所需的信息。
